# 企业级酒店客房管理系统 — 产品需求规格说明

| 文档属性 | 说明 |
|----------|------|
| 版本 | v1.0 |
| 状态 | 需求基线（待评审） |
| 适用对象 | 产品、业务、设计、研发对齐 |
| 范围说明 | 本文档仅描述业务与产品需求，不涉及技术实现 |

---

## 1. 文档目的

为「单体酒店、50 间以下规模」的 Web 端客房管理场景，定义第一版（MVP）必须交付的能力、业务规则、边界与后续版本规划，作为设计、开发与验收的统一依据。

---

## 2. 产品定位

### 2.1 目标用户与场景

- **酒店类型**：单体酒店，独立运营，客房规模 **50 间以下**。
- **使用场景**：前台日常接待（含原财务相关收款与交班）、店长运营监督、保洁房态协同。
- **「企业级」在本项目的含义**：
  - **权限精细**：岗位分工清晰，敏感操作需授权；
  - **可追溯**：订单全生命周期可审计（预订 → 入住 → 退房及关键变更）。

### 2.2 产品目标（MVP）

在不做 OTA 直连、餐饮、库存、会员与综合看板的前提下，支撑门店 **可日常营业** 的核心闭环：

**客房与房态 → 预订（手工）→ 入住（含 Walk-in）→ 在住管理 → 退房结算 → 保洁 → 交班对账**

### 2.3 明确不在 MVP 范围

| 类别 | 说明 |
|------|------|
| 会员管理 | 整体延后至后续版本 |
| 数据统计看板 | 独立看板延后；MVP 仅交班相关汇总 |
| OTA / 渠道直连 | 不做对接与房价同步 |
| 餐饮 POS | 不做 |
| 物资库存 | 不做 |
| 客人自助端 | 不做 |
| 集团多门店 | 不做跨店管理 |
| 续住 | 第一版不做（需续住时走退房 + 新入住或后续版本） |
| 钟点房、长住、团队批量、联房 | 第一版不做（换房/升房/降房除外） |

## 3. 角色与组织

### 3.1 系统角色（MVP）

| 角色 | 职责概要 |
|------|----------|
| **系统管理员** | 用户与角色维护；**下放改价等敏感权限**给指定账号/角色；系统参数维护 |
| **店长 / 运营经理** | 经营监督、报表查看、异常处理授权（如强制改房态等，按授权配置） |
| **前台** | 预订、排房、入住、在住变更、退房结算、收款、交班；**合并原财务前台职能**（收款汇总、待办事项） |
| **保洁** | 查看待打扫房间、标记打扫完成（第一版不分配到具体保洁员） |

> **变更说明**：原「财务」职能 **合并至前台**，不单独设财务登录角色。

### 3.2 用户账号规则

- 账号用于登录与权限控制，**不要求绑定真实姓名、工号等个人信息**。
- 账号命名与停用由管理员维护。
- 所有业务操作须关联 **当前登录账号**，用于审计追溯。

### 3.3 权限模型（业务要求）

- **菜单/功能级**：按角色控制可访问模块与操作入口。
- **敏感操作级**（须单独授权，可由管理员下放给前台等角色）：
  - 修改房价 / 折扣 / 订单金额；
  - 冲账、作废账单类操作（若 MVP 包含）；
  - 强制修改房态（绕过系统校验）；
  - 其他管理员可配置的敏感项。
- **数据范围**：MVP 以单店全局数据为主；若需「仅本班数据」在交班报表中体现，通过 **班次/交班记录** 关联，不做复杂多租户隔离。

## 4. 术语与状态

### 4.1 核心术语

| 术语 | 定义 |
|------|------|
| 房型 | 房间规格分类（如大床房、双床房），关联门市价与属性 |
| 客房 | 具体房号，归属房型、楼层等 |
| 预订 | 客人未来入住的订单，可预先排房 |
| 在住订单 | 已办理入住、尚未退房的住宿记录 |
| 营业日 | 按 **自然日 0 点** 切换，房费计入以营业日规则为准 |
| 交班 | 前台班次结束时的收款汇总与待办交接 |

### 4.2 房态（中等粒度）

| 房态 | 说明 |
|------|------|
| 空净 | 可售，可办理入住 |
| 预订锁定 | 已被预订占用，不可售给其他客人 |
| 预抵 | 当日预计到店（可与预订锁定并存展示逻辑由产品设计统一） |
| 在住 | 客人已入住 |
| 预离 | 当日预计退房 |
| 脏房 | 需保洁，触发打扫任务 |
| 维修 | 不可售，须记录原因与预计恢复时间 |

### 4.3 房态变更原则

- **系统强校验**：仅 **空净**（及业务规定的可售状态）允许新入住；防止同一房间重复分配给两批客人。
- 维修房、脏房、在住等 **不可直接新入住**，除非具备 **强制改房态** 权限的管理/授权操作（须留痕）。

## 5. 功能需求 — 客房信息管理（核心）

### 5.1 房型管理

- 维护房型名称、描述、**门市价**（默认单价，按「晚」或「天」计费规则需统一，见 8.2）。
- 维护房型属性：**床型、窗景、无烟、最大入住人数** 等（可扩展字段，MVP 至少支持列出的属性）。

### 5.2 客房管理

- 维护 **房号、所属房型、楼层**。
- 客房与房型属性继承关系：入住排房时以客房对应房型为准。
- 支持将客房置为 **维修**，必填：**维修原因、预计恢复时间**；恢复后变为空净或脏房（由业务流程定义，建议：维修完成 → 脏房或空净，若需保洁则经脏房流程）。

### 5.3 房态图 / 列表

- 支持按楼层、房型、房态筛选查看。
- 展示预抵、预离、预订锁定等中等粒度信息，便于前台一眼掌握当日态势。

---

## 6. 功能需求 — 预订管理

### 6.1 预订来源

- **仅支持前台/电话手工录入**，无 OTA、官网、渠道同步。

### 6.2 预订信息（MVP 最小集）

- 客人联系信息（**最小化采集**：至少满足联系与办理需要，如姓名、手机号；证件等 **第一版从简**，与客人档案策略一致，见 12.1）。
- 入住日期、离店日期、房型（或指定房号）、预订人/备注。
- 预订状态：待确认、已确认、已入住、已取消、已No-show（见 6.4）等（命名可调整，须覆盖生命周期）。

### 6.3 预先排房

- 预订时可 **预先指定房号**，该房进入 **预订锁定** 状态。
- 客人到店：**查询预订 → 确认排房 → 办理入住**（若预排房无冲突则一键确认）。

### 6.4 预订保留与释放（变更）

- **不再** 按保留时间 **自动取消** 并释放房态。
- 未到店或不再保留时，由 **前台手动释放** 预订及房态锁定（可标记 No-show 等状态，**不收取取消/No-show 罚金**）。
- **禁止超售**：预订排房时，系统 **硬拦截** 可售房数不足的情况。

### 6.5 预订变更与取消

- 支持改期、改房型、改排房、取消。
- **无系统自动化取消费规则**；取消不收取罚金（单体店简化策略）。

## 7. 功能需求 — 客人入住管理

### 7.1 Walk-in（无预订）

- 选择可售空净房 → 录入客人信息 → 确认房价 → 办理入住 → 房态变 **在住**。

### 7.2 预订入住

- 按预订单号/姓名/手机等查询 → 确认预排房或调整房号（调整须满足房态校验）→ 入住。

### 7.3 客人身份信息（第一版从简）

- 不做完整公安登记与多人逐人登记流程。
- MVP 以 **能完成联系与订单关联** 为准；证件等字段 **可选、从简**，后续版本加强。

### 7.4 押金

- **不管理押金/预授权**，仅记录与管理 **房费** 相关账务。

### 7.5 定价与改价

- 默认使用房型 **门市价**。
- 前台可在授权范围内 **打折或改价**；**改价权限由管理员下放**，无权限用户不可改价。
- 改价须进入 **审计日志**（谁、何时、原值、新值、关联订单）。

### 7.6 换房 / 升房 / 降房

- 支持在住期间更换房间或变更房型。
- **房费结算规则**：**整段入住期间按最终房型（及最终房价）统一重算**（非仅换房后分段价）。
- 换房须校验目标房态可售/可入住，并留痕。

### 7.7 提前入住 / 延迟退房

- **不配置系统自动半日/全日加收规则**。
- 由前台 **手工处理** 并在订单/备注中记录；费用若影响房费总额，通过改价或人工调整账单体现（须符合改价权限与审计要求）。

### 7.8 续住

- **MVP 不支持**；后续版本再定义续住与房态联动规则。

---

## 8. 功能需求 — 退房结算管理

### 8.1 费用项（MVP）

- **仅房费**（按预订/入住时的计费规则，含换房后整段重算结果）。
- 不含迷你吧、洗衣等额外消费（后续可扩展）。

### 8.2 计费规则（待业务确认默认值）

- 须在产品配置中明确：**按晚** 还是 **按天** 计费、半天规则等；MVP 建议默认 **按晚**，离店日当天是否计费需在参数或帮助中写清。
- 营业日按 **自然日 0 点**；跨夜房费归属以营业日规则为准。

### 8.3 结算与支付

- 退房前 **必须结清房费**（不存在未结清退房场景）。
- 支持 **分笔支付**（同一账单多种支付方式合计结清）。
- MVP 支付方式：**现金、微信、支付宝**（记录方式与金额，不对接支付通道细节）。
- **不管理押金抵扣**。

### 8.4 账单与审计

- 生成退房账单（明细：房费区间、单价、优惠、合计）。
- 结账、改价、作废（若有）须可追溯。

### 8.5 发票

- MVP **不** 对接税控/电子发票平台；若客人索要发票，可 **仅登记开票需求信息**（可选字段），供线下开票。

---

## 9. 功能需求 — 保洁打扫管理

### 9.1 任务触发

- 当房态变为 **脏房** 时，产生 **待打扫** 任务（退房后通常先将房态置为脏房，再触发；或由前台/保洁操作置脏）。

### 9.2 任务分配（MVP 简化）

- **不分配到具体保洁员**；仅维护房间级打扫状态。
- 保洁角色登录后可查看 **待打扫列表**（可按楼层筛选）。

### 9.3 完成与验收

- 保洁员标记 **打扫完成** 后，房态 **自动变为空净**，**无需** 主管查房或前台二次确认（MVP）。

### 9.4 与维修的区分

- 维修完成后的清洁流程：若业务要求保洁，应先进入脏房流程再变空净；须在操作指引中说明，避免跳过脏房导致状态不一致。

---

## 10. 功能需求 — 基础用户权限模块

### 10.1 用户与角色

- 管理员：创建/停用账号、分配角色、**配置敏感权限（含改价下放）**。
- 角色至少覆盖：管理员、店长、前台、保洁（前台合并原财务职能）。

### 10.2 登录与会话

- 账号密码登录（具体安全策略不在本文档）。
- 登出、会话超时等行为须可审计（可选记登录日志，最低要求见 11.2）。

### 10.3 操作日志（审计）

- **订单全生命周期** 关键节点须记录：创建/修改预订、入住、换房、改价、退房结账、取消、手动释放预订、强制改房态等。
- 每条日志至少包含：**操作人账号、时间、操作类型、关联订单/房号、变更摘要**；改价类须含 **变更前后金额**。

---

## 11. 功能需求 — 交班与统计（MVP subset）

### 11.1 交班对账（前台必做）

前台结班时生成/确认 **交班记录**，包含：

| 内容 | 要求 |
|------|------|
| 本班收款汇总 | 按支付方式（现金、微信、支付宝）汇总 |
| 待处理事项 | 未退房、待打扫、未释放预订等需交接项 |

- 须支持 **开班/结班** 或等效班次概念，以便关联本班收款（具体交互在 UI 阶段细化）。

### 11.2 店长统计（MVP 轻量）

- 不做独立「数据看板」模块。
- MVP 通过 **列表/简单汇总** 提供：**出租率（在住间数/可售间数）、房费营收（按日/区间）**，数据范围与交班、订单一致。
- **ADR / RevPAR、预订转化率、保洁效率** 等延后至看板版本。

## 12. 数据与合规（业务层）

### 12.1 客人档案

- **第一版从简**：可在订单上保存姓名、电话等；**不强制** 建立跨次入住的完整客人档案（后续版本可做历史档案）。

### 12.2 隐私与采集

- 遵循 **最小化采集**：仅收集办理预订、入住、联系所需字段。
- 证件信息 **非 MVP 必填**；若采集须有展示与留存说明（面向客人的告知由酒店自行负责）。

### 12.3 公安旅业登记

- MVP **不包含** 公安上传对接需求；若门店自行线下登记，系统不阻断。

---

## 13. 核心业务流程（摘要）

### 13.1 预订 → 入住

```
手工创建预订 → 可选预排房（房态：预订锁定）→ 客人到店查询预订
→ 确认房号与房价 → 办理入住（房态：在住）→ 预抵/预离随日期更新
```

### 13.2 退房 → 保洁 → 可售

```
发起退房 → 核对房费（整段计价）→ 收款结清 → 房态：脏房
→ 保洁待扫 → 保洁完成 → 房态：空净
```

### 13.3 预订未到

```
超过业务预期 → 前台手动释放预订与房态（可记 No-show，无罚金）
```

## 14. 业务规则汇总表

| 编号 | 规则 |
|------|------|
| BR-01 | 禁止超售，排房/入住时硬校验 |
| BR-02 | 非可售房态不可新入住，除非强制改房态（授权+留痕） |
| BR-03 | 预订保留由前台手动释放，不自动取消 |
| BR-04 | 取消/No-show 不收取罚金 |
| BR-05 | 改价须权限，管理员可下放给前台等 |
| BR-06 | 换房后房费按最终房型整段重算 |
| BR-07 | 退房须结清，仅房费，支持分笔支付 |
| BR-08 | 不管理押金 |
| BR-09 | 营业日按自然日 0 点 |
| BR-10 | 脏房触发保洁，完成后自动空净 |
| BR-11 | 维修房须填原因与预计恢复时间 |
| BR-12 | 账号不绑定姓名/工号，操作须记账号 |

## 15. 边缘场景与异常处理

| 场景 | 期望行为 |
|------|----------|
| 预排房房间维修中 | 入住前须提示冲突，引导换房或取消预排 |
| 换房目标房已被占 | 拦截并提示，须选其他空净房 |
| 在住中改价 | 允许（有权限），重算账单并审计 |
| 重复点击入住 | 幂等或拦截，防止重复在住单 |
| 手动释放预订 | 释放房态锁定，记录操作人；可关联 No-show 标记 |
| 强制改房态 | 仅授权用户，必填原因，强留痕 |
| 交班时有未结账单 | 应列入待处理，禁止结班忽略（或需店长确认，UI 阶段定） |
| 保洁标记完成但房实际未净 | 业务流程依赖门店管理；系统按状态机执行 |

## 17. 待确认项（Open Questions）

以下在 MVP 实施前建议业务方确认默认值：

| 编号 | 问题 |
|------|------|
| OQ-01 | 房费默认按「晚」还是「天」？离店当天是否计费？ |
| OQ-02 | 预订「预抵/预离」与「预订锁定」在房态图上的展示合并规则 |
| OQ-03 | 交班「开班/结班」是否强制：未结班是否允许下一账号登录操作 |
| OQ-04 | 半日房、钟点房是否在两年内纳入 — 当前为不做 |
| OQ-05 | 门市价调价是否需「生效日期」还是即时全局生效 |

## 18. 验收原则（MVP）

- 单店、50 间规模下，完成 **预订（预排房）→ 入住 → 换房重算 → 退房结清 → 脏房 → 保洁完成 → 空净** 全流程可走通。
- 无权限用户无法改价；管理员下放后前台可改价且可查日志。
- 超售、重复入住被系统拦截（强制除外且留痕）。
- 预订仅能通过前台 **手动释放**，无自动取消。
- 交班单含收款汇总与待办项；店长可查看出租率与房费营收简单汇总。
- 账号体系不依赖姓名/工号绑定，操作日志可定位到账号。

## 附录 A：需求澄清结论索引

- 酒店：单体，&lt;50 间

- 定价：门市价 + 授权改价；押金无；结账仅房费；支付现金/微信/支付宝  
- 保洁：脏房触发、不派人、完成即空净  
- 审计：订单全生命周期；隐私最小化
